## üîÑ Understanding Git Status Output

Let's start by understanding what `git status` tells us. This command shows the current state of our working directory and staging area.

```bash title="Check repository status"
git status
```

**Sample Output:**
```
On branch master
Changes not staged for commit:
  (use "git add <file>..." to update what will be committed)
  (use "git restore <file>..." to discard changes in working directory)
	modified:   second

no changes added to commit (use "git add" and/or "git commit -a")
```

**Understanding the Output:**
- **"On branch master"**: Shows which branch you're currently on
- **"Changes not staged for commit"**: Files that are modified but not yet added to staging area
- **"modified: second"**: The file `second` has been changed since the last commit
- **"no changes added to commit"**: Nothing is currently staged for commit

<Callout type="success">
‚úÖ **Git Status**: This is your most important command for understanding what Git sees. Always run `git status` before making any changes to understand your current state.
</Callout>

## üìù Making Changes and Staging Files

Let's see how changes move through Git's three areas:

```bash title="Make changes to a file"
nano first.txt
```

**Add some content:**
```text title="Content of first.txt"
hello

working

new changes
```

```bash title="Check status after making changes"
git status
```

**Output:**
```
On branch master
Changes not staged for commit:
  (use "git add <file>..." to update what will be committed)
  (use "git restore <file>..." to discard changes in working directory)
	modified:   first.txt
	modified:   second

no changes added to commit (use "git add" and/or "git commit -a")
```

**What This Shows:**
- Both `first.txt` and `second` have been modified
- Changes exist in working directory but not in staging area
- Git suggests using `git add` to stage changes or `git restore` to discard them

Now let's stage the changes:

```bash title="Stage all changes"
git add .
```

```bash title="Check status after staging"
git status
```

**Output:**
```
On branch master
Changes to be committed:
  (use "git restore --staged <file>..." to unstage)
	modified:   first.txt
	modified:   second
```

**What Changed:**
- Files moved from "Changes not staged" to "Changes to be committed"
- Both files are now in the staging area
- Git suggests using `git restore --staged` to unstage files

## üîÑ Git Reset: Unstaging Changes

The basic `git reset` command moves files from the staging area back to the working directory without deleting your changes.

```bash title="Reset staged changes"
git reset
```

**Output:**
```
Unstaged changes after reset:
M	first.txt
M	second
```

```bash title="Check status after reset"
git status
```

**Output:**
```
On branch master
Changes not staged for commit:
  (use "git add <file>..." to update what will be committed)
  (use "git restore <file>..." to discard changes in working directory)
	modified:   first.txt
	modified:   second

no changes added to commit (use "git add" and/or "git commit -a")
```

<Callout type="success">
‚úÖ **Git Reset Behavior**: The basic `git reset` command only affects the staging area. Your working directory changes remain intact. This is safe - you won't lose any work.
</Callout>

## ‚ö†Ô∏è Git Reset --Hard: Discarding Changes Completely

The `git reset --hard` command is more powerful and dangerous - it discards changes in both the staging area AND the working directory.

Let's stage our changes again first:

```bash title="Stage changes again"
git add .
```

```bash title="Verify files are staged"
git status
```

**Output:**
```
On branch master
Changes to be committed:
  (use "git restore --staged <file>..." to unstage)
	modified:   first.txt
	modified:   second
```

Now let's use the hard reset:

```bash title="Hard reset - WARNING: This discards all changes"
git reset --hard
```

**Output:**
```
HEAD is now at 4d0a85e update first.txt
```

```bash title="Check status after hard reset"
git status
```

**Output:**
```
On branch master
nothing to commit, working tree clean
```

Let's verify our changes are gone:

```bash title="Check file contents"
cat first.txt
```

**Output:**
```
hello

working
```

The "new changes" line we added is gone! Let's check the other file:

```bash title="Check second file"
cat second
```

**Output:** *(Empty file - no content)*

<Callout type="warning">
‚ö†Ô∏è **Warning**: `git reset --hard` permanently deletes changes that aren't committed. Use this command carefully! Always make sure you really want to discard your work before using it.
</Callout>

## üìÅ Introduction to .gitignore

The `.gitignore` file tells Git which files and directories to ignore. This is crucial for keeping your repository clean by excluding:

- Temporary files
- Build artifacts
- Environment-specific files
- Sensitive information
- IDE configuration files

Let's start with a practical example:

```bash title="Check current repository status"
git status
```

**Output:**
```
On branch master
nothing to commit, working tree clean
```

```bash title="Create a new file"
touch third.txt
```

```bash title="Add content to the file"
nano third.txt
```

**Content:**
```text title="Content of third.txt"
this is a new file
```

```bash title="Check status - Git sees the new file"
git status
```

**Output:**
```
On branch master
Untracked files:
  (use "git add <file>..." to include in what will be committed)
	third.txt

nothing added to commit but untracked files present (use "git add" to track)
```

**Understanding "Untracked files":**
- Git sees `third.txt` but isn't tracking it yet
- The file exists in your working directory but Git doesn't know about it
- You need to `git add` it to start tracking it

## üö´ Creating Your First .gitignore File

```bash title="Create .gitignore file"
touch .gitignore
```

```bash title="List all files including hidden ones"
ls -al
```

**Output:**
```
total 8
drwxr-xr-x. 3 centos9 centos9  84 Sep 23 12:47 .
drwxr-xr-x. 3 centos9 centos9  23 Sep 22 16:44 ..
-rw-r--r--. 1 centos9 centos9  15 Sep 23 12:41 first.txt
drwxr-xr-x. 8 centos9 centos9 183 Sep 23 12:46 .git
-rw-r--r--. 1 centos9 centos9   0 Sep 23 12:47 .gitignore
-rw-r--r--. 1 centos9 centos9   0 Sep 23 12:39 second
-rw-r--r--. 1 centos9 centos9  19 Sep 23 12:46 third.txt
```

**Notice the `.gitignore` file:**
- It starts with a dot, making it a hidden file
- Use `ls -al` to see hidden files (the `-a` flag shows all files)
- The `.git` directory is also hidden

```bash title="Add third.txt to .gitignore"
nano .gitignore
```

**Content of .gitignore:**
```text title=".gitignore content"
third.txt
```

```bash title="Check status after adding to .gitignore"
git status
```

**Output:**
```
On branch master
Untracked files:
  (use "git add <file>..." to include in what will be committed)
	.gitignore

nothing added to commit but untracked files present (use "git add" to track)
```

**What happened:**
- `third.txt` is no longer shown as untracked
- Git is now ignoring `third.txt` completely
- Only `.gitignore` itself is shown as untracked
- The `.gitignore` file is not ignored by default

<Callout type="success">
‚úÖ **Gitignore Effect**: Once a file is listed in `.gitignore`, Git acts as if it doesn't exist. The file is still there physically, but Git won't track it or show it in status.
</Callout>

## üéØ Gitignore Patterns: Wildcard Matching

### Ignoring File Extensions

Instead of ignoring individual files, you can ignore entire file types using patterns:

```bash title="Create multiple text files to test"
touch fourth.txt
```

```bash title="Check status - fourth.txt is not ignored yet"
git status
```

**Output:**
```
On branch master
Untracked files:
  (use "git add <file>..." to include in what will be committed)
	.gitignore
	fourth.txt

nothing added to commit but untracked files present (use "git add" to track)
```

```bash title="Update .gitignore to ignore all .txt files"
nano .gitignore
```

**New content:**
```text title="Updated .gitignore"
*.txt
```

**Understanding the `*` wildcard:**
- `*` matches any characters
- `*.txt` means "any file ending with .txt"
- This will ignore `third.txt`, `fourth.txt`, and any future `.txt` files

```bash title="Check status after wildcard pattern"
git status
```

**Output:**
```
On branch master
Untracked files:
  (use "git add <file>..." to include in what will be committed)
	.gitignore

nothing added to commit but untracked files present (use "git add" to track)
```

**Result:**
- Both `third.txt` and `fourth.txt` are now ignored
- Only `.gitignore` is still shown as untracked

### Testing with Different File Types

```bash title="Create Excel files to test"
touch test1.xlsx
touch test2.xlsx
```


**Output:**

```
-rw-r--r--. 1 centos9 centos9   0 Sep 23 13:01 test1.xlsx
```

- You can have multiple patterns in `.gitignore`
- Each pattern goes on its own line
- Empty lines are allowed for organization
- Both `.txt` and `.xlsx` files are now ignored

## üìÇ Ignoring Directories

You can also ignore entire directories and their contents:

```bash title="Create a directory with files"
mkdir build
```

```bash title="Navigate into the directory"
cd build/
```

```bash title="Create a file inside the directory"
touch hello.txt
```

```bash title="Check status from inside the build directory"
git status
```

**Output:**
```
On branch master
Untracked files:
  (use "git add <file>..." to include in what will be committed)
	../.gitignore

nothing added to commit but untracked files present (use "git add" to track)
```

**Understanding the output:**
- Git shows `../.gitignore` (the parent directory's .gitignore file)
- The `hello.txt` file isn't shown because `*.txt` is ignored
- But the build directory itself would be tracked if it had non-ignored files

```bash title="Go back to parent directory"
cd ..
```

```bash title="Check status from main directory"
git status
```

**Output:**
```
On branch master
Untracked files:
  (use "git add <file>..." to include in what will be committed)
	.gitignore

nothing added to commit but untracked files present (use "git add" to track)
```

Now let's test with a specific file ignored instead of the pattern:

```bash title="Change .gitignore to ignore specific file instead of pattern"
nano .gitignore
```

**Content:**
```text title="Ignore specific file instead of pattern"
third.txt

*.xlsx
```

```bash title="Check status with specific file ignored"
git status
```

**Output:**
```
On branch master
Untracked files:
  (use "git add <file>..." to include in what will be committed)
	.gitignore
	build/
	fourth.txt

nothing added to commit but untracked files present (use "git add" to track)
```

**What changed:**
- `third.txt` is still ignored (specific file)
- `fourth.txt` is now visible (*.txt pattern removed)
- `build/` directory is visible because it contains a `.txt` file that's not ignored by the specific pattern

Let's check what's in the build directory from Git's perspective:

```bash title="Check status from build directory"
cd build/
git status
```

**Output:**
```
On branch master
Untracked files:
  (use "git add <file>..." to include in what will be committed)
	../.gitignore
	./
	../fourth.txt

nothing added to commit but untracked files present (use "git add" to track)
```

**Understanding the paths:**
- `../.gitignore`: The .gitignore file in parent directory
- `./`: The current directory (build/)
- `../fourth.txt`: The fourth.txt file in parent directory

### Adding Directory to .gitignore

```bash title="Go back to main directory"
cd ..
```

```bash title="Add build directory to .gitignore"
nano .gitignore
```

**Content:**
```text title="Ignore directory and files"
third.txt

*.xlsx

build
```

```bash title="Check status after ignoring build directory"
git status
```

**Output:**
```
On branch master
Untracked files:
  (use "git add <file>..." to include in what will be committed)
	.gitignore
	fourth.txt

nothing added to commit but untracked files present (use "git add" to track)
```

**Result:**
- `build/` directory is no longer shown
- Git ignores the entire directory and everything inside it
- `fourth.txt` is still visible because it's not covered by our ignore patterns

<Callout type="success">
‚úÖ **Directory Ignoring**: When you add a directory name to `.gitignore`, Git ignores the entire directory and all its contents, regardless of what files are inside.
</Callout>

## ‚ö†Ô∏è Important .gitignore Rules

### 1. .gitignore Doesn't Affect Already Tracked Files

<Callout type="warning">
‚ö†Ô∏è **Important**: If a file is already being tracked by Git, adding it to `.gitignore` won't stop tracking it. You need to untrack it first with `git rm --cached filename`.
</Callout>

### 2. .gitignore Should Be Committed

```bash title="Add .gitignore to your repository"
git add .gitignore
git commit -m "Add .gitignore file"
```

The `.gitignore` file should be tracked so that all team members use the same ignore rules.

### 3. Global .gitignore

You can create a global `.gitignore` file for patterns you want to ignore across all your projects:

```bash title="Create global .gitignore"
git config --global core.excludesfile ~/.gitignore_global
```

## üö® Common Mistakes to Avoid

### 1. Using `git reset --hard` Without Thinking
```bash title="Dangerous - will lose all uncommitted work"
git reset --hard  # This deletes your changes permanently!
```

**Better approach:**
```bash title="Safe alternatives"
git status        # Check what you have first
git stash        # Save changes temporarily
# or
git add .        # Stage changes
git commit -m "WIP: work in progress"  # Save changes
```

### 2. Ignoring Files That Are Already Tracked
```bash title="This won't work if file is already tracked"
echo "config.json" >> .gitignore
```

**Correct approach:**
```bash title="First untrack, then ignore"
git rm --cached config.json    # Remove from tracking
echo "config.json" >> .gitignore  # Then ignore
git commit -m "Remove config.json from tracking"
```

### 3. Not Testing .gitignore Patterns
```bash title="Always test your .gitignore patterns"
git status  # Check if files are properly ignored
```

### ‚úÖ Remember These Points

1. **git reset**: Safely moves files from staging area to working directory
2. **git reset --hard**: Permanently discards changes in both staging and working directory
3. **.gitignore**: Controls which files Git ignores completely
4. **Patterns**: Use `*` for wildcards, directory names to ignore folders
5. **Safety**: Always check `git status` before using reset commands
